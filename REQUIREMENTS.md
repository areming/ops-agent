# REQUIREMENTS.md — ops-agent（暂定名）

> 阶段 1 产出物：需求澄清
> 状态：待你审阅确认。确认后进入阶段 2（技术选型）。

---

## 1. 一句话定位

一个**轻量级、可对话的运维助手**：像 claude code 一样在终端里交互，但操作对象是你的服务器。
小规模（1-3 台个人/小项目服务器），高度自主，能查、能改、能巡检、能自愈，并记住每台机器的来龙去脉。

---

## 2. 目标用户与场景

- **用户**：你本人（工程师），管理少量个人/小项目服务器。
- **典型场景**：
  - 在本地终端问"X 服务怎么挂了"，它去远端排查并给结论。
  - 说一句"帮我把 nginx 配置改成 XXX 并重载"，它去执行。
  - 后台自动巡检，发现问题尝试自愈，结果留痕，等你下次打开 CLI 时回看。
- **不是**：团队级的运维管控平台、多人协作系统、大规模集群编排工具。

---

## 3. 功能需求

### 3.1 交互形态
- **主入口：终端 CLI**，体验对标 claude code（对话式、流式输出、可追问）。
- 告警/自愈结果**不主动外推**，仅记录到本地；下次打开 CLI 时能回看历史。

### 3.2 部署形态
- **本地 CLI 客户端 + 每台被管服务器一个常驻 agent**（客户端/服务端模型）。
- agent 负责在所在机器上执行命令、采集信息、巡检、上报。

### 3.3 核心能力（按你勾选的全部四类）
1. **对话式排查与诊断**：查日志、看进程/资源、定位故障，由模型分析并给结论。
2. **对话式执行运维操作** ⭐（首版核心）：重启服务、部署、改配置、装软件等会改变系统状态的动作。
3. **批量/重复任务**：在多台机器上跑同样的命令或脚本。
4. **后台巡检 + 自愈**：agent 按节奏自动检查；发现异常先告警留痕，并在授权范围内尝试自动处理常见问题。

### 3.4 模型接入
- **多家厂商可切换**（Claude / OpenAI / 国产等），通过外接 API key 配置。
- 需要能在不同厂商/模型之间切换，而不是写死一家。

### 3.5 记忆
- **每台服务器一份知识档案**（类似 CLAUDE.md）：用途、装了什么、约束、踩过的坑，机器人会读取。
- **历史会话记忆**：记住之前做过哪些操作、结果如何，供后续对话和自愈参考。

---

## 4. 安全边界（关键，两条都要）

> 即使"高度自主"，以下两条是硬约束：

1. **不可逆操作必须拦截确认**：删数据、格式化、drop 数据库等不可恢复动作，再自主也要先问你。
2. **所有写操作完整可审计可回溯**：每次会改变系统状态的动作都有完整操作日志（做了什么、何时、结果），事后能查、尽量能改回来。

- 除上述两条外，默认**高度自主**：大部分情况自己决定并执行，仅在极危险时停下。

---

## 5. 非功能需求

- **轻量级 = 部署简单（最不能妥协的一条）**：agent 安装要尽量"单个二进制 / 一条命令"，不堆一大堆依赖。
- 资源占用、依赖轻重、概念简单度也是加分项，但优先级低于"部署简单"。

---

## 6. 明确的非目标（本项目不做）

- 不做团队/多用户/权限体系。
- 不做大规模集群编排、k8s 级调度。
- 首版不做主动外推告警（IM/邮件），告警仅本地留痕。
- 不做 Web/IM 等其它交互入口（首版只有 CLI）。

---

## 7. 首版优先级（MVP 起点）

**第一版只做好一件事：对话式执行运维操作。**
即"我说一个需求 → 它生成并执行命令搞定 → 全程可审计、危险操作拦截"。
其余能力（巡检自愈、批量、记忆、多模型切换）在此骨架之上分阶段叠加。

---

## 8. 需要你知道的张力 / 风险（我如实标注，未替你裁决）

1. **能力雄心 vs 轻量级**：巡检 + 自愈 + 高度自主 + 记忆，是一套不轻的能力集；和"部署简单/轻量"会拉扯。
   → 已用 MVP 优先级缓解：先做对话式执行，自愈等后置。阶段 4 roadmap 会据此排序。
2. **高度自主 + 自愈 的风险**：自动修复本身可能造成新问题。安全边界（第 4 节）是主要护栏，但自愈的"授权范围"具体怎么划，需在阶段 3 设计。
3. **告警只记本地 vs 监控的价值**：你选了简单优先，代价是出问题时不会实时通知你，得等你主动开 CLI。这是有意识的取舍，已记录。
